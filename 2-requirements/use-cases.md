## UC-001 — Просмотр расписания тренировок

**Акторы:** Клиент  
**Описание:** Клиент просматривает доступные слоты тренировок на 7 дней вперёд.  
**Предусловия:** Клиент авторизован в приложении.  
**Постусловия:** Клиент видит список слотов с информацией о дате, времени, зоне, инструкторе, свободных местах.  

**Основной сценарий:**
1. Клиент открывает приложение.
2. Система загружает и отображает слоты на 7 дней вперёд.
3. Клиент просматривает информацию о слотах (дата, время, зона, формат, инструктор, свободные места).

**Альтернативный сценарий (нет слотов):**
1. Система не находит слотов на выбранную дату.
2. Система отображает empty state («Пока нет доступных тренировок»).

**Исключения:** Ошибки загрузки данных из API.  
**Приоритет:** MVP  
**Источник:** [Раздел 4.1](../01-analysis/1-elicitation/domain-description.md#41-тренировка-слот), [Раздел 6](../01-analysis/1-elicitation/domain-description.md#6-основные-процессы-клиентская-перспектива)

## UC-002 — Запись на тренировку

**Акторы:** Клиент, Бэкенд/API  
**Описание:** Клиент записывается на выбранный слот тренировки.  
**Предусловия:** Клиент авторизован; слот доступен (есть свободные места); до начала ≥30 минут.  
**Постусловия:** Запись создана; клиент получил push-подтверждение.  

**Основной сценарий:**
1. Клиент выбирает слот из расписания.
2. Система проверяет доступность слота (свободные места, допуск инструктора для опытных тренировок).
3. Клиент выбирает снаряжение (своё / прокат по позициям / комбинация).
4. Система отображает разбивку стоимости (тренировка + прокат + итого).
5. Клиент подтверждает запись.
6. Система отправляет запрос на бронирование к бэкенду.
7. Бэкенд подтверждает наличие мест.
8. Система создаёт запись и отображает успех.
9. Система отправляет push-уведомление о подтверждении.

**Альтернативный сценарий (нет мест):**
1. Бэкенд возвращает отказ (мест нет).
2. Система отображает актуальное состояние слота (без кнопки записи).

**Альтернативный сценарий (нет допуска):**
1. Система определяет, что тренировка «опытная».
2. Система проверяет допуск инструктора.
3. При отсутствии допуска система блокирует запись и отображает сообщение.

**Исключения:** Ошибки API, таймауты.  
**Приоритет:** MVP  
**Источник:** [Раздел 4.1](../01-analysis/1-elicitation/domain-description.md#41-тренировка-слот), [Раздел 4.5](../01-analysis/1-elicitation/domain-description.md#45-запись-бронирование), [Раздел 6](../01-analysis/1-elicitation/domain-description.md#6-основные-процессы-клиентская-перспектива)

## UC-003 — Отмена записи клиентом

**Акторы:** Клиент, Бэкенд/API  
**Описание:** Клиент отменяет свою запись с учётом временных ограничений.  
**Предусловия:** У клиента есть активная запись.  
**Постусловия:** Запись отменена; место освобождено (если применимо).  

**Основной сценарий (>2 часов):**
1. Клиент открывает свои записи.
2. Клиент выбирает запись для отмены.
3. Система проверяет время до начала (>2 часов).
4. Система выполняет отмену.
5. Место освобождается.
6. Система отображает подтверждение отмены.

**Альтернативный сценарий (1–2 часа):**
1. Система определяет время до начала 1–2 часа.
2. Система отображает предупреждение о поздней отмене.
3. Клиент подтверждает отмену.
4. Место освобождается.

**Альтернативный сценарий (<1 часа):**
1. Система определяет время до начала <1 часа.
2. Система блокирует отмену и отображает сообщение о запрете.

**Исключения:** Ошибки API.  
**Приоритет:** MVP  
**Источник:** [Раздел 5.2](../01-analysis/1-elicitation/domain-description.md#52-отмена-клиентом), [Раздел 6](../01-analysis/1-elicitation/domain-description.md#6-основные-процессы-клиентская-перспектива)

## UC-004 — Отмена тренировки скалодромом

**Акторы:** Владелец/администратор (вне приложения), Клиент, Бэкенд/API  
**Описание:** Скалодром отменяет тренировку; клиент получает уведомление и предложение альтернативы.  
**Предусловия:** Тренировка запланирована; есть активные записи.  
**Постусловия:** Запись переведена в статус «Отменена скалодромом»; клиент уведомлён.  

**Основной сценарий:**
1. Владелец отменяет слот в админке (вне клиентского приложения).
2. Бэкенд обновляет статус слота.
3. Система отображает слот с пометкой об отмене.
4. Система отправляет обязательное push-уведомление клиенту.
5. Система отображает причину отмены и извинения.
6. Система предлагает ближайший свободный слот с теми же параметрами.
7. При наличии оплаты статус меняется на «возврат».

**Альтернативный сценарий (нет альтернативы):**
1. Система не находит подходящего альтернативного слота.
2. Система информирует клиента, что замену нужно искать самостоятельно.

**Исключения:** Ошибки доставки push.  
**Приоритет:** MVP  
**Источник:** [Раздел 5.3](../01-analysis/1-elicitation/domain-description.md#53-отмена-скалодромом), [BR-016–BR-022](business-requirements.md#53-отмена-скалодромом)

## UC-005 — Регистрация клиента

**Акторы:** Клиент, Бэкенд/API  
**Описание:** Новый клиент регистрируется в приложении по телефону.  
**Предусловия:** Клиент не зарегистрирован.  
**Постусловия:** Клиент авторизован; профиль создан.  

**Основной сценарий:**
1. Клиент вводит номер телефона.
2. Система запрашивает обязательные данные: ФИО, телефон, дата рождения.
3. Клиент заполняет поля.
4. Система отправляет данные на бэкенд.
5. Бэкенд создаёт профиль.
6. Система авторизует клиента.

**Исключения:** Ошибки валидации, ошибки API.  
**Приоритет:** MVP  
**Источник:** [Раздел 4.4](../01-analysis/1-elicitation/domain-description.md#44-клиент), [BR-030](business-requirements.md#br-030--регистрация-по-телефону)

## UC-006 — Подтверждение согласия на риск

**Акторы:** Клиент, Бэкенд/API  
**Описание:** Клиент подтверждает согласие на риск при первой записи.  
**Предусловия:** Клиент делает первую запись в жизни.  
**Постусловия:** Согласие зафиксировано; запись разрешена.  

**Основной сценарий:**
1. Система определяет, что это первая запись клиента.
2. Система отображает текст согласия на риск.
3. Клиент подтверждает согласие.
4. Система сохраняет подтверждение.
5. Запись продолжается.

**Исключения:** Клиент отказывается — запись прерывается.  
**Приоритет:** MVP  
**Источник:** [Таблица MVP vs Post-MVP](../01-analysis/1-elicitation/domain-description.md#mvp-vs-post-mvp), [BR-031](business-requirements.md#br-031--согласие-на-риск)

## UC-007 — Получение push-напоминаний

**Акторы:** Клиент, Бэкенд/API  
**Описание:** Клиент получает push-напоминания о предстоящей тренировке.  
**Предусловия:** У клиента есть активная запись.  
**Постусловия:** Клиент уведомлён.  

**Основной сценарий:**
1. Бэкенд инициирует отправку напоминаний (за сутки и за N часов).
2. Система получает и отображает push-уведомление.
3. Клиент видит напоминание.

**Исключения:** Push не доставлен (ошибка ОС).  
**Приоритет:** MVP  
**Источник:** [BR-026](business-requirements.md#br-026--типы-push-уведомлений), [BR-027](business-requirements.md#br-027--время-напоминаний), [BR-028](business-requirements.md#br-028--обязательные-уведомления)

## UC-009 — Изменение проката после записи

**Акторы:** Клиент, Бэкенд/API  
**Описание:** Клиент изменяет выбранный прокат после создания записи.  
**Предусловия:** У клиента есть активная запись; нужные позиции есть в свободном фонде.  
**Постусловия:** Прокат обновлён; сумма пересчитана.  

**Основной сценарий:**
1. Клиент открывает детали записи.
2. Клиент выбирает изменение проката.
3. Система проверяет наличие нужных позиций в свободном фонде.
4. Система обновляет прокат.
5. Система пересчитывает сумму.
6. Система отображает обновлённые данные.

**Альтернативный сценарий (нет позиций):**
1. Система не находит нужные позиции в фонде.
2. Система отображает сообщение о недоступности.

**Исключения:** Ошибки API.  
**Приоритет:** MVP  
**Источник:** [Раздел 4.5](../01-analysis/1-elicitation/domain-description.md#45-запись-бронирование), [BR-040](business-requirements.md#br-040--изменение-проката-после-записи)

## UC-010 — Получение бейджа постоянного клиента

**Акторы:** Клиент, Бэкенд/API  
**Описание:** Клиент получает бейдж «постоянный клиент» после достижения N посещений.  
**Предусловия:** Клиент завершил N тренировок (N — конфигурируемый параметр).  
**Постусловия:** Бейдж отображается в профиле; применена скидка/бонус.  

**Основной сценарий:**
1. Бэкенд отслеживает количество посещений.
2. При достижении N бэкенд присваивает статус.
3. Система отображает бейдж в профиле клиента.
4. Система применяет скидку/бонус.

**Исключения:** Нет.  
**Приоритет:** MVP  
**Источник:** [Таблица MVP vs Post-MVP](../01-analysis/1-elicitation/domain-description.md#mvp-vs-post-mvp), [BR-032](business-requirements.md#br-032--постоянный-клиент)

## UC-011 — Оценка инструктора (post-MVP)

**Акторы:** Клиент, Бэкенд/API  
**Описание:** Клиент оценивает инструктора звёздами после завершённой тренировки.  
**Предусловия:** Тренировка завершена; прошло не более 1–2 суток; тренировка не отменена скалодромом.  
**Постусловия:** Оценка сохранена; средняя оценка инструктора обновлена.  

**Основной сценарий:**
1. Система определяет завершённую тренировку.
2. Система отправляет push-приглашение к оценке (отключаемое).
3. Клиент открывает форму оценки.
4. Клиент выставляет оценку (1–5 звёзд).
5. Система сохраняет оценку.
6. Средняя оценка инструктора обновляется.

**Альтернативный сценарий (отмена скалодромом):**
1. Система определяет, что тренировка отменена скалодромом.
2. Система блокирует оценку и отображает сообщение.

**Исключения:** Окно оценки истекло.  
**Приоритет:** Post-MVP  
**Источник:** [Раздел 4.7](../01-analysis/1-elicitation/domain-description.md#47-оценка-инструктора-post-mvp), [BR-033–BR-035](business-requirements.md#57-оценка-инструктора-post-mvp)

## UC-012 — Просмотр средней оценки инструктора (post-MVP)

**Акторы:** Клиент  
**Описание:** Клиент просматривает среднюю оценку инструктора перед записью.  
**Предусловия:** Инструктор имеет оценки.  
**Постусловия:** Клиент видит рейтинг инструктора.  

**Основной сценарий:**
1. Клиент просматривает слот в расписании.
2. Система отображает среднюю оценку инструктора рядом с его именем.
3. Клиент учитывает оценку при выборе слота.

**Исключения:** Нет оценок — отображается заглушка.  
**Приоритет:** Post-MVP  
**Источник:** [Раздел 4.3](../01-analysis/1-elicitation/domain-description.md#43-инструктор), [BR-035](business-requirements.md#br-035--средняя-оценка-инструктора)
