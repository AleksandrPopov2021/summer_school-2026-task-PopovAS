## Обзор

Логика выбора снаряжения (своё / прокат по позициям), проверки доступности в фонде слота и локального расчёта разбивки стоимости на SCR-005 и SCR-007. Данные черновика используются в LOGIC-005 и LOGIC-010.

### User Story

> Как клиент, я хочу выбрать нужное снаряжение и видеть итоговую стоимость,
> чтобы принять решение о записи на тренировку.

### Бизнес-ценность

- Гибкий выбор: своё + прокат (BR-002)
- Прозрачная разбивка стоимости до подтверждения (FR-011)
- Предотвращение выбора недоступных позиций проката

## Описание логики

### Шаг 1: Загрузка справочника

При открытии формы — [`listRentalEquipmentTypes`](../api/openapi.yaml) (кэшируется в памяти сессии). Позиции: `shoes`, `harness`, `helmet`, `chalk`.

### Шаг 2: Загрузка доступности слота

[`getSlotRentalAvailability`](../api/openapi.yaml) возвращает `available_quantity` по каждой позиции. Чекбокс неактивен при `available_quantity = 0`.

### Шаг 3: Правила выбора (BR-002)

- `uses_own_equipment = true` — клиент использует своё снаряжение
- `rental_lines` — выбранные позиции проката с `quantity` (обычно 1)
- Комбинация своё + прокат разрешена: оба флага могут быть активны

Минимум одно из: своё снаряжение или хотя бы одна позиция проката.

### Шаг 4: Локальный расчёт стоимости (FR-011)

```
training_amount = slot.training_price
rental_amount = Σ (unit_price × quantity) для каждой rental_line
  где unit_price = equipment_type.default_price (или rental_tariff слота если задан)
discount_amount = client.loyalty_discount применительно к training_amount (если is_loyal_client)
total_amount = training_amount + rental_amount - discount_amount
```

Расчёт **локальный для превью**; итоговые суммы в записи приходят из API в `PaymentInfo` после создания.

### Шаг 5: Сохранение черновика

При каждом изменении обновляется `booking_draft`:

```json
{
  "slot_id": "uuid",
  "uses_own_equipment": true,
  "rental_lines": [
    { "equipment_type_id": "uuid", "quantity": 1 }
  ]
}
```

## API запросы

### GET /rental-equipment-types — `listRentalEquipmentTypes`

**Триггер:** Открытие SCR-005

**Обработка ответа:**

| Результат | Действие |
|-----------|----------|

| Поле | Описание |
|------|----------|
| `Authorization` | `Bearer {access_token}` |

**Обработка ответа:**

| Результат | Действие |
|-----------|----------|
| Успех (200) | Обновить доступность чекбоксов |
| Ошибка 404 | Снек, возврат на SCR-004 |

## Критерии приёмки

| ID | Критерий |
|----|----------|
| AC-001 | **Дано** справочник загружен, **Когда** открывается SCR-005, **Тогда** отображаются 4 позиции проката |
| AC-002 | **Дано** `available_quantity = 0` для туфель, **Когда** форма отрисована, **Тогда** чекбокс туфель заблокирован |
| AC-003 | **Дано** выбраны туфли и «Своё снаряжение», **Когда** расчёт выполнен, **Тогда** `uses_own_equipment = true` и rental_lines содержит туфли |
| AC-004 | **Дано** выбор изменён, **Когда** пользователь видит разбивку, **Тогда** отображаются тренировка, прокат и итого (FR-011) |
| AC-005 | **Дано** клиент — постоянный, **Когда** расчёт выполнен, **Тогда** в разбивке видна скидка |
| AC-006 | **Дано** выбор сохранён, **Когда** пользователь уходит с SCR-005, **Тогда** `booking_draft` содержит актуальный выбор |

## Обработка ошибок

| Тип ошибки | Контекст | Действие |
|------------|----------|----------|
| Нет доступных позиций | Все quantity = 0 | Предложить только «Своё снаряжение» |
| Устаревшая доступность | Перед POST | Повторный getSlotRentalAvailability |
