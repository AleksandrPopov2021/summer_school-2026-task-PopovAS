## Обзор

Логика отмены активной записи клиентом с учётом `cancellation_policy` из API. Определяет доступность кнопки отмены, уровень предупреждения на SCR-008 и вызов DELETE /bookings.

### User Story

> Как клиент, я хочу отменить запись на тренировку с понятными правилами,
> чтобы освободить место, если планы изменились.

### Бизнес-ценность

- Соблюдение временных окон отмены (BR-010, BR-011, BR-012)
- Предупреждение о поздней отмене для снижения злоупотреблений (BR-014)
- Синхронизация с бэкендом — окончательное решение за API

## Описание логики

### Шаг 1: Интерпретация `CancellationPolicy`

| `warning_level` | `can_cancel` | UI на SCR-006/007 | Текст SCR-008 | BR |
|-----------------|--------------|-------------------|---------------|-----|
| `none` | true | Кнопка активна | «Вы уверены, что хотите отменить запись?» | BR-010 |
| `late_cancellation` | true | Кнопка активна | «Внимание! Отмена за 1–2 часа…» | BR-011 |
| `forbidden` | false | Кнопка скрыта/заблокирована | — | BR-012 |

Поля `minutes_until_start` — для отображения в подсказке («до начала осталось X минут»).

### Шаг 2: Экран подтверждения

SCR-008 получает `booking_id` и `warning_level`. Вариант текста определяется уровнем предупреждения.

### Шаг 3: Вызов API

[`cancelBooking`](../api/openapi.yaml) — DELETE без тела. Бэкенд освобождает место (BR-010, BR-011).

### Шаг 4: Обновление списка

После 200 — инвалидировать кэш записей, обновить SCR-006. Слот в расписании получит +1 место при следующем LOGIC-003.

### Шаг 5: Перенос не поддерживается

Отмена не предлагает перенос — только новая запись (BR-013).

## API запросы

### DELETE /bookings/{bookingId} — `cancelBooking`

**Триггер:** Подтверждение на SCR-008

**Headers:**

| Поле | Описание |
|------|----------|
| `Authorization` | `Bearer {access_token}` |

**Параметры/Body:**

| Параметр | Тип | Описание | Значение/Источник |
|----------|-----|----------|-------------------|
| `bookingId` | uuid | ID записи | Навигация |

**Обработка ответа:**

| Результат | Действие |
|-----------|----------|
| Загрузка | Spinner на SCR-008 |
| Успех (200) | SCR-006, снек об успехе |
| Ошибка 403 `CANCELLATION_FORBIDDEN` | Снек, обновить `cancellation_policy` |
| Ошибка 409 | Снек «Запись уже отменена» |
| Ошибка 404 | Снек, SCR-006 |
| Ошибка 5xx | Снек «Произошла ошибка. Попробуйте позже» |
| Ошибка сети | Снек «Нет соединения. Проверьте подключение к интернету» |

### GET /bookings/{bookingId} — `getBooking`

**Триггер:** После 403 для актуализации политики

**Обработка ответа:**

| Результат | Действие |
|-----------|----------|
| Успех (200) | Обновить UI кнопки отмены |

## Критерии приёмки

| ID | Критерий |
|----|----------|
| AC-001 | **Дано** `warning_level = none`, **Когда** пользователь нажимает «Отменить», **Тогда** SCR-008 с обычным текстом |
| AC-002 | **Дано** `warning_level = late_cancellation`, **Когда** открывается SCR-008, **Тогда** отображается предупреждение BR-011 |
| AC-003 | **Дано** `warning_level = forbidden`, **Когда** запись на SCR-007, **Тогда** кнопка отмены заблокирована |
| AC-004 | **Дано** пользователь подтвердил отмену, **Когда** DELETE возвращает 200, **Тогда** запись исчезает из SCR-006 |
| AC-005 | **Дано** DELETE возвращает 403, **Когда** ошибка обработана, **Тогда** снек с `message`, политика обновлена |
| AC-006 | **Дано** `can_cancel = false`, **Когда** список записей отрисован, **Тогда** кнопка «Отменить» не отображается |

## Обработка ошибок

| Тип ошибки | Контекст | Действие |
|------------|----------|----------|
| CANCELLATION_FORBIDDEN | DELETE 403 | Обновить policy, заблокировать кнопку |
| Гонка: отмена скалодромом | 409 | Снек, обновить статус записи |
| Нет сети | DELETE | Снек, остаться на SCR-008 |
