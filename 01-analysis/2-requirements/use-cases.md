# Use Cases — Скалодром «Вертикаль»

## UC-001: Просмотр расписания тренировок

**ID:** UC-001  
**Название:** Просмотр расписания тренировок  
**Акторы:** Клиент  
**Описание:** Клиент просматривает доступные тренировки на 7 дней вперёд.  
**Приоритет:** Высокий (MVP)  
**Источник:** domain-description.md, раздел 4.1, MVP таблица

**Предусловия:**
- Клиент открыл мобильное приложение

**Основной сценарий:**
1. Система отображает слоты тренировок на ближайшие 7 дней по умолчанию
2. Клиент выбирает дату для просмотра
3. Система отображает доступные слоты на выбранную дату
4. Для каждого слота система показывает: время начала, длительность, зону/формат, инструктора, свободные места (осталось X из Y), прокатный тариф, адрес

**Альтернативные сценарии:**
- 3a. Если слотов нет: система отображает empty state «Пока нет доступных тренировок»
- 3b. Если слот отменён скалодромом: система отображает слот с пометкой об отмене

**Постусловия:**
- Клиент ознакомился с доступными тренировками

## UC-002: Запись на тренировку

**ID:** UC-002  
**Название:** Запись на тренировку  
**Акторы:** Клиент, Бэкенд/API  
**Описание:** Клиент записывается на выбранную тренировку.  
**Приоритет:** Высокий (MVP)  
**Источник:** domain-description.md, раздел 4.1, раздел 4.5, раздел 5.1

**Предусловия:**
- Клиент авторизован в приложении
- Клиент выбрал подходящий слот

**Предусловия для слота:**
- До начала слота >= 30 минут
- Есть свободные места (> 0)
- Для опытных тренировок: у клиента есть допуск инструктора

**Основной сценарий:**
1. Клиент выбирает слот для записи
2. Система проверяет ограничения (время, места, допуск)
3. Клиент выбирает снаряжение: своё или прокат по позициям (скальные туфли, страховочная система, каска, магнезия)
4. Система отображает разбивку стоимости: тренировка + прокат + итого
5. При первой записи: клиент подтверждает согласие на риск
6. Клиент подтверждает запись
7. Система отправляет запрос на бронирование в бэкенд
8. Бэкенд подтверждает наличие мест и создаёт запись
9. Система отображает подтверждение записи
10. Система отправляет push-уведомление о подтверждении (если не отключено)

**Альтернативные сценарии:**
- 2a. Если до начала < 30 минут: система блокирует запись, отображает сообщение
- 2b. Если мест нет (0): система скрывает кнопку записи
- 2c. Если опытная тренировка без допуска: система блокирует запись, отображает сообщение
- 8a. Если бэкенд отказал (места заняты): система отображает актуальное состояние слота, запись не создана

**Постусловия:**
- Запись создана и отображается в списке активных записей клиента

## UC-003: Отмена записи клиентом

**ID:** UC-003  
**Название:** Отмена записи клиентом  
**Акторы:** Клиент, Бэкенд/API  
**Описание:** Клиент отменяет свою активную запись.  
**Приоритет:** Высокий (MVP)  
**Источник:** domain-description.md, раздел 5.2

**Предусловия:**
- У клиента есть активная запись
- До начала тренировки > 1 часа

**Основной сценарий:**
1. Клиент открывает список своих записей
2. Клиент выбирает запись для отмены
3. Система проверяет время до начала тренировки
4. **Вариант A (> 2 часов):**
   - Система позволяет отмену без предупреждения
   - Клиент подтверждает отмену
   - Система отправляет отмену в бэкенд
   - Место освобождается
5. **Вариант B (1–2 часа):**
   - Система отображает предупреждение
   - Клиент подтверждает отмену
   - Система отправляет отмену в бэкенд
   - Место освобождается

**Альтернативные сценарии:**
- 3a. Если < 1 часа до начала: система блокирует отмену, отображает сообщение «Отмена запрещена менее чем за 1 час»

**Постусловия:**
- Запись переведена в статус «Отменено клиентом»
- Место освобождено для других клиентов

## UC-004: Отмена тренировки скалодромом

**ID:** UC-004  
**Название:** Отмена тренировки скалодромом  
**Акторы:** Скалодром (администратор), Клиент, Бэкенд/API  
**Описание:** Скалодром отменяет тренировку, клиенты получают уведомления.  
**Приоритет:** Высокий (MVP)  
**Источник:** domain-description.md, раздел 5.3

**Предусловия:**
- Существует активная запись клиента на слот
- Администратор инициирует отмену в существующей инфраструктуре

**Основной сценарий:**
1. Администратор отменяет слот в системе скалодрома
2. Система переводит все записи на слот в статус «Отменена скалодромом»
3. Система фиксирует причину отмены из фиксированного списка
4. Система добавляет краткие извинения
5. Система отправляет обязательный push всем затронутым клиентам
6. Для оплаченных записей: система меняет статус оплаты на «возврат»
7. Слот остаётся в списке с пометкой об отмене
8. Система предлагает ближайший свободный слот с теми же параметрами (зона, сложность, инструктор, прокат)

**Альтернативные сценарии:**
- 8a. Если нет альтернативного слота: клиент ищет замену самостоятельно

**Постусловия:**
- Все клиенты уведомлены
- Повторная запись на этот слот запрещена
- Статусы оплаты обновлены (при необходимости)

## UC-005: Получение push-напоминаний

**ID:** UC-005  
**Название:** Получение push-напоминаний  
**Акторы:** Клиент, Система уведомлений  
**Описание:** Клиент получает напоминания о предстоящей тренировке.  
**Приоритет:** Высокий (MVP)  
**Источник:** domain-description.md, раздел 5.5 (BR-027, BR-028)

**Предусловия:**
- У клиента есть активная запись
- Настройки push разрешены (напоминания обязательны)

**Основной сценарий:**
1. За сутки до тренировки: система отправляет push-напоминание
2. В день тренировки за N часов (N — конфигурируемый параметр): система отправляет второе push-напоминание
3. Клиент получает уведомления на устройстве

**Альтернативные сценарии:**
- Нет (напоминания обязательны, не отключаются)

**Постусловия:**
- Клиент напомнен о предстоящей тренировке

## UC-007: Получение статуса постоянного клиента

**ID:** UC-007

**Акторы:** Клиент, Система лояльности  
**Описание:** Клиент получает бейдж и скидку после N посещений.  
**Приоритет:** Средний (MVP)

- Клиент имеет историю посещённых тренировок

2. При достижении N посещений: система присваивает статус «постоянный клиент»
3. Система отображает бейдж в профиле клиента

**Альтернативные сценарии:**
- Нет

**Постусловия:**
- Клиент имеет статус постоянного клиента с соответствующими привилегиями

## UC-008: Изменение выбора проката

**ID:** UC-008  
**Название:** Изменение выбора проката  
**Акторы:** Клиент, Бэкенд/API  
**Описание:** Клиент изменяет выбор снаряжения после записи.  
**Приоритет:** Средний (MVP)  
**Источник:** domain-description.md, раздел 4.5

**Предусловия:**
- У клиента есть активная запись с выбранным прокатом

5. Система обновляет запись с новой разбивкой стоимости

**Альтернативные сценарии:**
- 3a. Если позиций нет в фонде: система отображает сообщение, изменение невозможно

- Выбор проката обновлён

## UC-009: Оценка инструктора (post-MVP)

**ID:** UC-009  
**Название:** Оценка инструктора  
**Акторы:** Клиент  
**Описание:** Клиент оценивает инструктора после завершённой тренировки.  
**Приоритет:** Низкий (Post-MVP)  
**Источник:** domain-description.md, раздел 4.7, раздел 5.7 (BR-033, BR-034)

**Предусловия:**
- Тренировка завершена
- Прошло не более 1–2 суток после завершения
- Тренировка не была отменена скалодромом

**Основной сценарий:**
1. Система отправляет push-приглашение оценить инструктора (если не отключено)
2. Клиент открывает форму оценки
3. Клиент выставляет оценку звёздами 1–5 (без текста)
4. Система сохраняет оценку
5. Средняя оценка инструктора обновляется

**Альтернативные сценарии:**
- 1a. Если тренировка отменена скалодромом: оценка недоступна
- 3a. Если прошло более 2 суток: оценка недоступна

**Постусловия:**
- Оценка сохранена
- Средняя оценка инструктора обновлена (отображается другим клиентам)

2. Система отображает среднюю оценку инструктора (звёзды 1–5)
3. Клиент учитывает рейтинг при выборе слота

**Альтернативные сценарии:**
- 2a. Если оценок нет: система не отображает рейтинг

- Клиент информирован о рейтинге инструктора

## UC-011: Управление настройками уведомлений

**ID:** UC-011  
**Название:** Управление настройками уведомлений  
**Акторы:** Клиент  
**Описание:** Клиент настраивает получаемые push-уведомления.  
**Приоритет:** Средний (MVP)  
**Источник:** domain-description.md, раздел 5.5 (BR-028, BR-029)

**Предусловия:**
- Клиент авторизован в приложении

**Основной сценарий:**
1. Клиент открывает настройки уведомлений
2. Клиент видит список типов уведомлений:
   - Подтверждение записи (отключаемое)
   - Приглашение к оценке (отключаемое)
   - Напоминания (обязательное, не отключается)
   - Отмена скалодромом (обязательное, не отключается)
3. Клиент переключает доступные для изменения уведомления

**Альтернативные сценарии:**
- Нет (обязательные уведомления не отключаются)

**Постусловия:**
- Настройки уведомлений сохранены

## UC-012: Просмотр статуса оплаты

**ID:** UC-012  
**Название:** Просмотр статуса оплаты  
**Акторы:** Клиент  
**Описание:** Клиент просматривает сумму и статус оплаты записей.  
**Приоритет:** Высокий (MVP)  
**Источник:** domain-description.md, раздел 4.8, раздел 5.4 (BR-023, BR-025)

**Предусловия:**
- У клиента есть активная или завершённая запись

**Основной сценарий:**
1. Клиент открывает детали записи
2. Система отображает разбивку стоимости: тренировка + прокат + итого
3. Система отображает статус оплаты: не оплачено, оплачено, возврат
4. Клиент видит информацию для офлайн-оплаты (наличные, перевод)

**Альтернативные сценарии:**
- Нет (онлайн-оплата вне скоупа MVP)

**Постусловия:**
- Клиент информирован о статусе платежа
