# Нефункциональные требования (Non-Functional Requirements)

На основе документа: [domain-description.md](../01-analysis/1-elicitation/domain-description.md)

---

## NFR-001 — Мобильный-first UX
**Формулировка:** Интерфейс приложения должен быть оптимизирован для использования на смартфоне (основной канал использования клиентами в зале).  
**Приоритет:** MVP  
**Источник:** [Раздел 1](../01-analysis/1-elicitation/domain-description.md#1-контекст-бизнеса), [Раздел 8](../01-analysis/1-elicitation/domain-description.md#8-нефункциональные-ожидания)

---

## NFR-002 — Быстрый доступ к слотам
**Формулировка:** Пользователь должен иметь быстрый доступ к просмотру слотов по дате и своим записям (минимум кликов).  
**Приоритет:** MVP  
**Источник:** [Раздел 8](../01-analysis/1-elicitation/domain-description.md#8-нефункциональные-ожидания)

---

## NFR-003 — Актуальность данных из API
**Формулировка:** Данные о свободных местах и прокатном фонде должны получать актуальные данные из бэкенд-API в реальном времени.  
**Приоритет:** MVP  
**Источник:** [Раздел 8](../01-analysis/1-elicitation/domain-description.md#8-нефункциональные-ожидания), [Раздел 2](../01-analysis/1-elicitation/domain-description.md#2-границы-текущей-поставки)

---

## NFR-004 — Понятные сообщения об ошибках
**Формулировка:** Система должна отображать понятные сообщения при конфликте бронирования, запрете отмены, отсутствии допуска инструктора.  
**Приоритет:** MVP  
**Источник:** [Раздел 8](../01-analysis/1-elicitation/domain-description.md#8-нефункциональные-ожидания)

---

## NFR-005 — Обязательность push-уведомлений
**Формулировка:** Push-уведомления об отмене скалодромом и напоминания должны быть обязательными (не отключаются пользователем).  
**Приоритет:** MVP  
**Источник:** [Раздел 8](../01-analysis/1-elicitation/domain-description.md#8-нефункциональные-ожидания), [BR-028](business-requirements.md#br-028--обязательные-уведомления)

---

## NFR-006 — Язык интерфейса
**Формулировка:** Язык интерфейса в MVP — только русский.  
**Приоритет:** MVP  
**Источник:** [Раздел 1](../01-analysis/1-elicitation/domain-description.md#1-контекст-бизнеса), [Раздел 8](../01-analysis/1-elicitation/domain-description.md#8-нефункциональные-ожидания), [Таблица MVP vs Post-MVP](../01-analysis/1-elicitation/domain-description.md#mvp-vs-post-mvp)

---

## NFR-007 — Конфигурируемость параметров
**Формулировка:** Параметры системы (N часов для напоминания, N посещений для лояльности, N нарушений для санкций) должны быть конфигурируемыми без изменения кода приложения.  
**Приоритет:** MVP  
**Источник:** [Раздел 8](../01-analysis/1-elicitation/domain-description.md#8-нефункциональные-ожидания)

---

## NFR-008 — Производительность на мобильных устройствах
**Формулировка:** Приложение должно обеспечивать быстрое время отклика на смартфонах при просмотре расписания и выполнении операций записи/отмены.  
**Приоритет:** MVP  
**Источник:** [Раздел 1](../01-analysis/1-elicitation/domain-description.md#1-контекст-бизнеса) (ограничение: основной канал — смартфон)

---

## NFR-009 — Интеграция с бэкендом как источником истины
**Формулировка:** Бэкенд является единственным источником истины для данных: слоты, зоны/форматы, инструкторы, прокатный фонд; клиентское приложение только потребляет данные.  
**Приоритет:** MVP  
**Источник:** [Раздел 2](../01-analysis/1-elicitation/domain-description.md#2-границы-текущей-поставки)

---

## NFR-010 — Обработка ошибок API
**Формулировка:** Приложение должно корректно обрабатывать ответы API об отказе в бронировании и отображать актуальное состояние слота.  
**Приоритет:** MVP  
**Источник:** [Раздел 2](../01-analysis/1-elicitation/domain-description.md#2-границы-текущей-поставки), [BR-003](business-requirements.md#br-003--подтверждение-бронирования-бэкендом)

---

## NFR-011 — Срок поставки MVP
**Формулировка:** Система должна быть готова к началу нового сезона (~2 месяца от начала разработки).  
**Приоритет:** MVP  
**Источник:** [Раздел 1](../01-analysis/1-elicitation/domain-description.md#1-контекст-бизнеса)

---

## NFR-012 — Ограниченность бюджета
**Формулировка:** Разработка ведётся в условиях ограниченного бюджета → приоритет функциональности MVP, отложение post-MVP сценариев.  
**Приоритет:** MVP  
**Источник:** [Раздел 1](../01-analysis/1-elicitation/domain-description.md#1-контекст-бизнеса)

---

## NFR-013 — Независимость от деталей бэкенда
**Формулировка:** Детали аутентификации, маппинга статусов и внутренней модели бэкенда не обсуждаются с заказчиком и не влияют на клиентское приложение.  
**Приоритет:** MVP  
**Источник:** [Раздел 2](../01-analysis/1-elicitation/domain-description.md#2-границы-текущей-поставки)

---
